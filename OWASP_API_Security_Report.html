Task 7: Harden JWT Configuration

Overview

Fixed the API8 Weak Authentication vulnerability by implementing comprehensive JWT hardening. The original implementation used weak secrets, excessively long token lifetimes, and lacked proper claim validation.

Vulnerability Description

OWASP API Security Category: API8:2023 - Security Misconfiguration (Weak Authentication)

Original Issues:

  1. Weak Secret Key: JWT secret only 44 characters, vulnerable to brute force
  2. Long Token TTL: 30-day token validity provided large attack window
  3. No Issuer/Audience: Tokens lacked iss and aud claims
  4. Weak Algorithm: HS256 (256-bit) instead of HS512 (512-bit)

Changes Made

1. Strong Key Generation

@Value("${app.jwt.secret}")
private String secret;

private Key getSigningKey() {
    return Keys.hmacShaKeyFor(secret.getBytes());
}

2. Short Token Lifetime

@Value("${app.jwt.ttl-seconds:900}")
private long ttlSeconds;  // 15 minutes default

3. Issuer and Audience Claims

private static final String ISSUER = "OWASP_API_VULN_LAB";
private static final String AUDIENCE = "APP_USERS";

return Jwts.builder()
        .setIssuer(ISSUER)
        .setAudience(AUDIENCE)
        .signWith(getSigningKey(), SignatureAlgorithm.HS512)
        .compact();

4. Strict Validation

Jwts.parserBuilder()
        .setSigningKey(getSigningKey())
        .requireIssuer(ISSUER)
        .requireAudience(AUDIENCE)
        .build()
        .parseClaimsJws(token);

Security Benefits

  • Strong Keys: Proper key derivation with HS512
  • Reduced Attack Window: 15 minutes vs 30 days
  • Token Scope Limitation: Issuer/audience prevent reuse
  • Stronger Algorithm: HS512 (512-bit)

API5: Broken Function Level Authorization (BFLA)

Overview

Fixed the API5 Broken Function Level Authorization vulnerability by implementing role-based access control (RBAC) for administrative endpoints. This prevents regular users from accessing privileged functionality.

Vulnerability Description

OWASP API Security Category: API5:2023 - Broken Function Level Authorization

Original Issue:

The original implementation did not properly enforce role-based authorization. While authentication was required, any authenticated user could potentially access administrative endpoints such as:

  • /api/admin/metrics - System metrics and monitoring data
  • /api/admin/users - User management functions
  • Other administrative operations

This is a critical security flaw where horizontal privilege escalation could occur - regular users gaining admin privileges.

Fix Implemented

SecurityConfig.java - Role-Based Authorization

Added explicit role checks using Spring Security hasRole method:

http.authorizeHttpRequests(reg -> reg
    // Public endpoints
    .requestMatchers("/api/auth/login", "/api/auth/signup").permitAll()
    
    // FIX: Admin endpoints require ADMIN role
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    
    // All other API endpoints require authentication
    .requestMatchers("/api/**").authenticated()
    .anyRequest().authenticated()
);

How It Works

  1. JWT Token Contains Role: When users log in, their JWT token includes a role claim (USER or ADMIN)
  2. JWT Filter Extracts Role: The JWT filter extracts the role and creates a Spring Security authority
  3. Spring Security Enforces: Before processing requests to /api/admin/**, Spring Security checks for ROLE_ADMIN
  4. Access Denied: If the user lacks the ADMIN role, they receive HTTP 403 Forbidden

Security Benefits

  • Prevents Privilege Escalation: Regular users cannot access admin functions
  • Clear Separation of Duties: Admin vs. user roles enforced at framework level
  • Consistent Authorization: All /api/admin/** endpoints automatically protected
  • Fail-Secure: Denies access by default if role is missing

API10: Unsafe Consumption of APIs

Overview

While this application does not currently consume external third-party APIs, we have documented comprehensive security guidelines for future integrations to prevent API10 vulnerabilities.

Vulnerability Description

OWASP API Security Category: API10:2023 - Unsafe Consumption of APIs

API10 addresses risks when an application integrates with external APIs without proper security controls. This includes:

  • No Input Validation: Trusting data from external APIs without validation
  • Missing TLS/HTTPS: Consuming APIs over unencrypted connections
  • Credential Exposure: Hardcoding API keys in source code
  • No Rate Limiting: Not handling rate limits from external APIs
  • Redirect Following: Blindly following redirects without validation
  • Excessive Trust: Assuming external API responses are always safe

Security Guidelines for Future API Integrations

1. Always Use HTTPS/TLS

// GOOD: Enforce HTTPS for external API calls
RestTemplate restTemplate = new RestTemplate();
String apiUrl = "https://api.example.com/data"; // HTTPS only

2. Validate External API Responses

// GOOD: Validate response before processing
@Valid
public class ExternalApiResponse {
    @NotNull
    @Size(min=1, max=100)
    private String name;
}

3. Store API Keys Securely

// GOOD: Load from environment variables
@Value("${external.api.key}")
private String apiKey;

4. Implement Timeout and Circuit Breakers

// GOOD: Set timeouts to prevent hanging
SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
factory.setConnectTimeout(5000);  // 5 seconds
factory.setReadTimeout(10000);    // 10 seconds
